# Abstract

ASES defines the engineering process that AI coding agents must follow when participating in professional software development. It establishes requirements for artifacts, quality gates, traceability, documentation, verification, and maintainability.

The goal is not to restrict AI capabilities but to ensure that AI-assisted development produces software that is maintainable, verifiable, traceable, and production-ready.

---

# Purpose

This standard answers:

> What constitutes good software engineering when AI agents are part of the development team?

---

# Scope

ASES applies to:

- Any AI coding agent (MiMoCode, Claude Code, Codex, Cursor, Cline, OpenHands, etc.)
- Any programming language
- Any technology stack
- Any project size

ASES does not apply to:

- Prototype-only experiments
- Throwaway scripts
- Learning exercises

---

# Core Principles

## Principle 1: Developer Accountability

Even when assisted by an AI agent, the developer holds ultimate responsibility for the quality, functionality, performance, security, and maintainability of the code they commit.

AI is a tool. "The AI did it" is not a valid excuse.

---

## Principle 2: Understand and Verify

Code generated by AI must not be accepted without understanding. Before integrating any code, its functionality and implications must be verified through review and testing.

Blind acceptance is prohibited.

---

## Principle 3: Security First

Sensitive information must never be exposed to unapproved AI agents. Even with approved tools, remain vigilant about data leakage and strictly adhere to security policies.

---

## Principle 4: Code Quality and Consistency

AI-generated code must meet the same coding standards, architectural patterns, and quality requirements as human-written code.

The developer is responsible for ensuring compliance.

---

## Principle 5: Human-Led Design

Core system design, architectural decisions, and critical business logic must be led by humans. AI assists but does not replace architectural judgment.

---

## Principle 6: AI Awareness

Be aware of AI limitations: hallucinations, biases, outdated knowledge, lack of contextual understanding. Continuously adapt to changing technology and build team capabilities.

---

## Principle 7: Epistemic Humility

AI agents must recognize their limitations and ask for help when uncertain. No agent should pretend to know something it doesn't, or proceed confidently when uncertain.

When uncertain, ask. When unsure, verify. When wrong, correct.

---

## Principle 8: Transparency

AI agents must provide explainable decisions and maintain audit trails. Every significant decision should be justified, and the reasoning behind it documented.

"The AI did it" is not an explanation. Decisions must be traceable.

# Engineering Lifecycle

ASES defines a 10-phase engineering lifecycle. Every feature must complete all applicable phases.

```text
Phase 1  — Specification
Phase 2  — Requirement Review
Phase 3  — Impact Analysis
Phase 4  — Solution Design
Phase 5  — Controlled Implementation
Phase 6  — Verification
Phase 7  — Documentation
Phase 8  — Maintainability Audit
Phase 9  — Technical Debt Review
Phase 10 — Final Review
```

Each phase has:

- **Required artifacts** — What must be produced
- **Quality gates** — What must be verified before proceeding
- **Exit criteria** — What defines completion

# Quality Gates

Every phase transition requires passing quality gates. Gates are defined in `quality-gates.md`.

Gate severity levels:

| Severity | Behavior |
|----------|----------|
| Error | Blocks progression. Must be resolved. |
| Warning | Flags risk. Should be addressed. |
| Info | Informational. No blocking. |

# Definition of Done

A feature is complete only when:

## Specification

- [ ] Feature Specification exists
- [ ] User Story approved
- [ ] Acceptance Criteria defined
- [ ] Definition of Done specified
- [ ] Dependencies identified
- [ ] Risks identified

## Design

- [ ] Impact Assessment complete
- [ ] Implementation Plan approved
- [ ] Task Breakdown complete
- [ ] ADR created (if applicable)

## Implementation

- [ ] Code implemented
- [ ] Code reviewed
- [ ] Coding standards satisfied
- [ ] No unauthorized refactoring

## Testing

- [ ] Unit Tests passed
- [ ] Integration Tests passed
- [ ] E2E Tests passed (if applicable)
- [ ] Acceptance Criteria verified

## Documentation

- [ ] User Manual updated (if applicable)
- [ ] API Documentation updated (if applicable)
- [ ] Release Notes updated

## Quality

- [ ] No Critical Bugs
- [ ] No High Severity Bugs
- [ ] Maintainability Report generated
- [ ] Technical Debt reviewed

## Governance

- [ ] Traceability complete
- [ ] Policy checks passed
- [ ] Quality gates passed
- [ ] Reviewer sign-off obtained

---

# Definition of Failure

A feature fails if:

- Any Acceptance Criterion is not met
- Any required test is missing or failing
- Documentation is not updated
- Quality gates are not passed
- Scope was exceeded without approval
- Critical or High severity bugs remain

---

# Traceability

Every engineering artifact must be traceable. The traceability chain is:

```text
Epic
   ↓
Feature Specification
   ↓
Tasks
   ↓
Source Code
   ↓
Unit Tests
   ↓
Integration Tests
   ↓
E2E Tests
   ↓
Documentation
   ↓
Release
```

Detailed traceability requirements are defined in `traceability.md`.

---

# Testing Standards

Testing requirements are defined in `testing-standards.md`.

Key requirements:

- Every Acceptance Criterion must have at least one test
- Unit test coverage must meet project threshold
- E2E tests required for user-facing features
- Tests must be meaningful, not just coverage numbers

---

# Documentation Standards

Documentation requirements are defined in `documentation-standards.md`.

Key requirements:

- Documentation is part of the product
- Documentation must be updated with code changes
- Documentation freshness must be validated
- Required documentation depends on feature type

---

# Security Standards

Security requirements are defined in `security-standards.md`.

Key requirements:

- Sensitive data must not be hardcoded
- Agent security settings must be verified
- Dependencies must be reviewed
- Input validation must be implemented

---

# Agent Interaction Standards

How humans should interact with AI agents is defined in `agent-interaction-standards.md`.

Key requirements:

- Plan before implementation
- Verify before acceptance
- Manage context aggressively
- Use subagents for investigation
- Maintain checkpoints for rollback

---

# Compliance

Compliance with ASES is validated by the Agent Governance Engine (AGE).

Projects must:

1. Reference this standard in their policy
2. Configure quality gates per their needs
3. Define technology-specific rules
4. Maintain traceability matrices
